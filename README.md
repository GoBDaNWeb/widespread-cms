Админ-панель для онлайн-магазина одежды: каталог товаров, дашборд с аналитикой и авторизация.

Проект-витрина того, как я проектирую и пишу фронтенд: от архитектуры и прод-подобной авторизации на httpOnly-cookie до работы с кэшем и тестов на бизнес-логику. Каждое решение в проекте выбрано осознанно и под конкретную задачу - ниже я объясняю, почему именно так.

> **Статус по проекту.** Готовы авторизация, дашборд и весь модуль товаров - создание, редактирование, удаление, фильтрация, пагинация и массовые операции. Остальные разделы постепенно добавляются. Архитектура с самого начала заложена так, чтобы новый модуль рос по уже отлаженному шаблону, а не требовал переписывать половину проекта.

## Стек и почему именно он

**React 19 и TypeScript в strict-режиме.** Типизация сквозная, от ответа API до пропсов компонента, без `any` - ошибки контракта ловятся на сборке, а не в рантайме у пользователя.

**Маршрутизация - TanStack Router.** Типобезопасный, валидирует search-параметры и, главное, даёт грузить данные ещё до рендера страницы - на этом построена защита роутов (см. раздел про авторизацию). Заодно нативно дружит с TanStack Query.

**Состояние - осознанно разделено.** Серверное держит **TanStack Query**, клиентское - **Zustand**. Это не вкусовщина: смешивать их - источник целого класса багов с рассинхроном, поэтому копий серверных данных в Zustand нет в принципе (подробнее - в разделе про архитектуру). Zustand, а не Context, потому что UI-стейт (модалки, выделенные строки) меняется часто, а перерисовывать поддерево каждый раз не хочется.

**Формы - React Hook Form + Zod.** Одна Zod-схема даёт и рантайм-валидацию, и выведенный из неё тип формы - источник правды один, дублировать типы руками не нужно.

Остальное - под конкретные задачи: стили на **Tailwind v4**, графики дашборда на **Recharts**, rich-text описания на **Tiptap**, drag-and-drop сортировка картинок на **dnd-kit**, тесты на **Vitest + Testing Library**.

Бэкенд я написал отдельно на FastAPI, он живёт в своём репозитории - [widespread-backend](https://github.com/BogdanOgn/widespread-backend).

## Архитектура: FSD

Проект построен на [Feature-Sliced Design](https://feature-sliced.design/) - методологии со слоями, нарезанными по бизнес-смыслу, а не по типам файлов.

Почему именно она, а не привычные альтернативы. Группировка по типам (`components/`, `hooks/`, `utils/`) на дистанции разрастается: код одной фичи размазан по всему дереву, и непонятно, что можно трогать, не задев соседей. Плоские feature-папки решают это наполовину - но без явных правил фичи начинают тянуть друг друга, и через полгода в этом будет проблематично разобраться. FSD задаёт оба ограничения сразу: и группировку по смыслу, и направление зависимостей между слоями. Цена - больше папок и порог входа в саму методологию; для проекта-витрины, который должен расти по предсказуемому шаблону, размен честный.

```
src/
├── app/        # инициализация: провайдеры, роутер, точка входа
├── pages/      # страницы - собирают виджеты и фичи под конкретный маршрут
├── widgets/    # самостоятельные блоки UI (Sidebar, ResourceLayout)
├── features/   # пользовательские сценарии (логин, создание и правка товара)
├── entities/   # бизнес-сущности (user, product, brand, category, size)
└── shared/     # ядро: ui-kit, api-клиент, конфиг, утилиты
```

На практике из этого вытекают три вещи, которые мне в нём и нужны.

Зависимости идут строго в одну сторону: слой видит только то, что ниже него (`features` > `entities` > `shared`). Циклов не возникает в принципе, это режется ещё на уровне импортов. Наружу каждый слайс отдаёт только свой публичный API через `index.ts` - внутреннюю структуру модуля можно перетряхивать сколько угодно, потребители этого не заметят. А новая сущность/фича - это просто новая папка по тому же шаблону, без рефакторинга соседей и без конфликтов в git, когда над проектом работает не один человек.

<img width="399" height="644" alt="Image" src="https://github.com/user-attachments/assets/487cb972-2068-4d50-8239-09e2be17da07" />

Отдельно подчёркну одно решение, потому что оно экономит кучу багов: **серверное и клиентское состояние я не смешиваю**. Всё, что приходит с бэкенда, живёт только в TanStack Query - никаких копий серверных данных в локальном сторе и вечной возни с их синхронизацией. В Zustand остаётся ровно UI-стейт: какие модалки открыты, какие строки таблицы выделены.

## Авторизация

Это та часть, которую я делал максимально близко к тому, как авторизацию стоит делать в проде, а не «сохранили токен в localStorage и забыли».

Токены (access и refresh) лежат в **httpOnly-cookie** и из JavaScript недоступны - значит, их нельзя украсть через XSS. Фронтенд вообще не знает их значений: запросы уходят с `withCredentials`, а cookie браузер подставляет сам.

Самое интересное - обновление сессии. Если запрос словил `401`, axios-интерсептор по-тихому обновляет токен и **повторяет исходный запрос**, так что пользователь ничего не замечает. Тонкость в том, что refresh устроен как single-flight: когда параллельно отвалилось сразу несколько запросов, обновление выполняется **один раз**, а остальные ждут в очереди и доисполняются уже с новым токеном - без шторма одинаковых refresh-запросов на сервер.

```
запрос → 401 ┐
запрос → 401 ┼─► refresh выполняется ОДИН раз
запрос → 401 ┘         │
                       ├─ успех  → все запросы повторяются
                       └─ провал → событие auth:logout → редирект на /auth
```

Если обновиться не удалось, HTTP-слой просто кидает событие `auth:logout`. Так он остаётся развязан с роутингом и не тащит за собой зависимость от роутера - кто слушает это событие и куда редиректит, его не касается.

Доступ к страницам решается в **loader'ах роутов**, ещё до рендера:

```ts
export const requireUser = async (qc: QueryClient) => {
	const user = await qc.ensureQueryData(authQueries.me());
	if (!user) throw redirect({ to: ROUTES.AUTH });
};
```

`ensureQueryData` либо берёт пользователя из кэша, либо дозапрашивает - поэтому защищённый контент не мелькает на экране до проверки. Эта же `me()`-query отвечает и за персист сессии: после F5 пользователь восстанавливается из cookie, логиниться заново не нужно. Зеркальный гард `requireGuest` наоборот уводит уже залогиненного со страницы `/auth`.

При входе ответ сервера сразу кладётся в кэш, при выходе кэш чистится целиком - чтобы данные предыдущего пользователя случайно не утекли в следующую сессию.

## Кэш и query-ключи

Ключи запросов я не разбрасываю строками по компонентам, а собираю в типизированные фабрики рядом с сущностью. Устроены они иерархически - от общего к частному:

```ts
export const productQueries = {
	all: () => ({ queryKey: ['products'] as const }),
	lists: () => ({ queryKey: [...productQueries.all().queryKey, 'list'] as const }),
	list: filters =>
		queryOptions({
			queryKey: [...productQueries.lists().queryKey, filters] as const,
			queryFn: () => getProducts(filters),
			placeholderData: keepPreviousData
		}),
	product: id =>
		queryOptions({
			queryKey: [...productQueries.all().queryKey, id],
			queryFn: () => getProduct(id)
		}),
	stats: () =>
		queryOptions({
			queryKey: [...productQueries.all().queryKey, 'stats'],
			queryFn: getProductsStats
		})
};
```

Иерархия нужна ради инвалидации. После любой мутации товара я инвалидирую `productQueries.lists()` - и обновляются **все** списки с любыми наборами фильтров, но при этом не трогается кэш отдельного товара и статистики. Ключ описан в одном месте, так что опечаток в строках просто негде взяться. А `keepPreviousData` на списке делает пагинацию и фильтрацию плавными: пока грузится новая страница, на экране остаётся предыдущая, без мигания.

Сам `QueryClient` настроен под админку: `staleTime` пять минут, чтобы справочники не дёргались на каждый фокус окна, и ретраи отключены на `401/403` - это не сетевой сбой, повторять такой запрос смысла нет.

## Фильтры живут в URL

Состояние каталога - страница, фильтры, сортировка - я держу в URL, а не в локальном стейте. Ссылку на отфильтрованный каталог можно скопировать, кинуть другому человеку или положить в закладки, и она откроется ровно в том же виде.

Сырые параметры из строки браузера проходят через `validateSearch` роутера - там они нормализуются и валидируются: строки конвертируются в числа, мусор отбрасывается, пагинация получает дефолты. `?page=abc` не уронит приложение, а спокойно превратится в `page: 1`.

## Тесты

Тестирую бизнес-логику, а не разметку. Под покрытием - мутации товаров (create / update / delete и обе массовые операции) с проверкой, что инвалидируются именно нужные ключи; нормализация search-параметров со всеми граничными случаями; и Zod-схемы форм.

```sh
npm test            # прогон
npm run test:watch  # watch-режим
npm run test:coverage
```

## Качество кода

Чтобы в `main` не утекал мусор, проверки навешаны на git-хуки через **husky**. На `pre-commit` отрабатывает **lint-staged**: ESLint с автофиксом и Prettier гоняются только по застейдженным файлам, а не по всему проекту - быстро. На `commit-msg` стоит **commitlint** с conventional-commits, так что история коммитов остаётся однородной и пригодной для автоген-ченджлога. Всё это запускается локально само, помнить про линтеры не нужно.

## Запуск

Нужен **Node.js >= 20.19** (рекомендую LTS 22) и npm.

```sh
git clone https://github.com/BogdanOgn/widespread-cms.git
cd widespread-cms
npm i
npm run dev            # дев-сервер на http://localhost:5173
```

Запросы на `/api` проксируются на бэкенд - по умолчанию `http://127.0.0.1:8000`, адрес переопределяется через `VITE_API_URL`. Бэкенд поднимается из [отдельного репозитория](https://github.com/BogdanOgn/widespread-backend).

Остальные команды:

```sh
npm run build     # прод-сборка (tsc -b + vite build)
npm run preview   # локальный предпросмотр прод-сборки
npm run lint      # статический анализ (ESLint)
npm test          # прогон тестов
```
